# Session Optimize

把会话中的失败和返工转化为待批准的项目改进或明确移交。不要为了产出复盘而制造问题，未经用户批准严禁修改任何内容。

## 1. 确定范围并收集证据

- 分析自上次 `/session-optimize` 以来的会话；首次调用时分析当前可见的完整会话。
- 优先使用会话原文、工具结果、验证记录和实际 diff；需要核实项目事实时读取当前项目。
- 将每个候选问题记为：`触发情境 → 实际行为 → 影响 → 期望行为`。
- 只有具体失败、用户纠正、返工、验证不通过或明显成本浪费才算问题。正常探索、个人偏好和未经验证的猜测不算。
- 合并同根问题，保留最能解释因果关系的证据。

## 2. 归因与定级

先区分症状和根因。将根因归入且只归入一个主类别：

- `项目实现`：当前项目的代码、测试、配置或运行行为有缺陷。
- `工具本身`：工具、连接器或平台能力异常。
- `工具使用`：选择、参数、调用顺序或结果解释错误。
- `知识缺口`：完成任务所需的知识、项目文档或说明错误或缺失；文档与实现不一致也归入此类。
- `背景输入不足`：目标、约束或项目事实未提供且无法自行发现。
- `流程/决策`：计划、验证、边界判断或协作方式不当。
- `外部环境`：权限、网络、第三方服务、运行环境或其他角色负责的条件。

用反事实检查归因：若信息充分且用法正确，问题是否仍会发生？不要把工具误用归为工具故障，也不要把项目外故障归为项目实现。

严重程度只用：

- `高`：阻塞任务、造成错误结果，或带来数据与安全风险。
- `中`：导致返工、明显降质或重复浪费。
- `低`：轻微影响效率或清晰度，可择机优化。

## 3. 决定申请修改还是移交

首次复盘只诊断和提案。读取与检查可以直接进行；写文件、改配置、执行会改变状态的命令或操作外部系统都属于修改，必须先批准。

项目内候选修改只有在以下条件全部满足时才可提交给用户审批：

1. 目标位于当前项目内；
2. 修改局部、可回退且不改变未经确认的产品行为；
3. 会话证据能说明修改与问题之间的因果关系；
4. 能用现有环境验证结果；
5. 不需要其他角色、额外权限或外部状态配合。

对每项候选修改先展示：

- 精确目标文件或状态；
- 拟议改动及理由；
- 风险与影响范围；
- 修改后的验证方式。

随后明确请求用户批准，可让用户逐项或整批批准。沉默、推断、以前的泛化授权和“看起来合理”都不算批准。

收到针对该方案的明确批准后，才可修改获批范围。修改前重新读取目标并遵守项目指令；保留用户已有改动，实施最小有效修复并运行相关验证。若出现新目标、范围扩大、额外副作用或行为变化，立即停止并重新申请批准。不要顺手清理无关内容。

以下情况只移交，不进入修改审批：其他 skill 或本 skill 的规则、全局配置、依赖或插件安装、外部系统、Git 提交或推送、破坏性操作、需要产品决策的行为变化，以及证据不足或无法验证的改动。本 skill 自身的问题交给 `refine-skill`。

知识缺口若能通过当前项目内已有依据补全项目文档，可按上述门槛申请修改；通用模型知识缺口只给出补充来源或下次指令建议。

## 4. 输出

不要默认创建复盘文件。按以下顺序回复：

1. **总体结论**：一句话说明发现多少问题、提出多少项待批准修改；没有实质问题时明确写“本轮无需优化”。
2. **问题清单**：用表格列出 `严重程度 | 类别 | 证据 | 根因 | 处置`。
3. **待批准项目改进**：列出目标、拟议改动、理由、风险和验证方式，并明确请求批准；没有则写“无”。
4. **待移交事项**：逐项给出负责角色、未修改原因、建议方案、验证标准和可直接复制的移交说明；没有则写“无”。
5. **下次会话输入**：只列能避免本轮信息缺口的可复制背景说明；没有则写“无”。

用户随后批准执行时，最终回复另列 **已批准并完成的改进**，说明批准范围、实际改动和验证结果。

所有结论都必须能回指证据。不要重复叙述已经成功的步骤，也不要把建议伪装成已完成修改。
