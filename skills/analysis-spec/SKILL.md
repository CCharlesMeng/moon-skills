# 分析规格

## 概述

这个 skill 把一个模糊的需求描述转化为可执行、可验证、可追溯的分析包。

内部有五个 phase，按顺序执行：

| Phase | 做什么 | 产出 |
| --- | --- | --- |
| 0 — 准备 | 需求分级 + 上下文预热 + 确定检查清单 | 工作模式、相关 refs、激活的领域检查清单 |
| A — 摸底 | 摸清当前行为 | 现状记录 |
| B — 自学习 | 读取已有知识，产出三色状态表 | 🔴 / 🟡 / 🟢 置信度表 |
| C — 澄清 | 多轮红绿灯澄清 | 用户确认和工作假设 |
| D — 规格 | 产出目标行为和风险清单 | `analysis-spec.md` |

## 硬门禁

1. **不要跳过摸底。** 不了解现状就写目标行为，是在对空气做设计。
2. **不要跳过自学习。** 没有先学就直接问用户，是在浪费用户时间。
3. **不要把推测写成事实。** 🟡 项必须标明推理过程，让用户自行判断。
4. **不要在澄清未收敛时进入规格。** 所有 🔴 必须变 🟢 才能继续。
5. **不要在规格中写实现方案。** analysis-spec 只描述行为和约束，不决定技术选型。

## 何时使用

适用场景：

- 收到一个前端 / 后端 / 混合需求，需要在开始切片和实现前对齐目标行为和风险
- 需求来源可以是 issue、口头沟通、设计稿或产品文档

不适用场景：

- 仓库尚未初始化（先用 `initialize`）
- 当前是事故排查（用 `immune-debug`）
- 已经有清晰的 analysis-spec，只需要切片（直接用 `slice-plan`）

## 工作流

### Phase 0 — 准备

在分析开始前，做三件事。

**第一步：需求分级**

判断这个需求属于哪种工作模式：

| 工作模式 | 适用场景 | 后续影响 |
| --- | --- | --- |
| `patch-lite` | 局部文案、样式、叶子组件小行为修正 | 所有 phase 可压缩，澄清默认最小粒度 |
| `feature-slice` | 一个完整用户路径或页面能力的新增/调整 | 完整流程，澄清默认按需粒度 |
| `migration-strict` | 跨模块改造、共享组件/状态迁移 | 完整流程 + 额外兼容约束，澄清默认全量粒度 |

判断标准：

- `patch-lite` 需要**全部满足**：只影响单个叶子组件、不改路由/共享组件/状态/权限/埋点、当前行为 5 条以内讲清
- `migration-strict` **任一成立**即升级：影响多个模块、修改共享层、需要兼容期/flag/灰度、说不清影响面
- 拿不准就按更重的处理，只允许从轻往重升级

**第二步：上下文预热**

- 仓库没有 `.project-context/profile.yaml` → 先用 `initialize`
- 仓库已有 `.project-context` → 用 `sync-context` 为目标模块做预热
- 确认目标模块、已有 refs、当前 uncertainty

**第三步：确定领域检查清单**

领域检查清单是一组在 Phase D 中按需装配的检查项，用于确保分析包覆盖了特定领域的关键问题。

内置 4 个默认检查清单：

| 检查清单 | 激活信号 |
| --- | --- |
| **前端** | 需求涉及 UI、交互、布局、客户端体验 |
| **后端** | 需求涉及 API、数据、权限、可靠性 |
| **集成** | 需求跨两个以上边界或存在联动契约 |
| **风险记忆** | 需求触达高危模块或历史事故（除 `patch-lite` 外默认开启） |

完整检查项和自定义扩展方式见下方"领域检查清单"章节。

退出门禁：

- 工作模式已确定
- 目标模块可定位
- 不确定项已显式列出
- 已确定激活哪些领域检查清单

---

### Phase A — 摸底

回答"现在是什么"。

动作：

- 记录当前用户路径与页面状态
- 找出入口点：路由、组件树、状态源、API、埋点
- 识别共享组件、共享 hooks、设计系统、样式依赖
- 识别权限、加载、空态、错误态、回退逻辑
- 识别现有测试、验证方式和验证缺口
- 对迁移类需求补兼容承诺：哪些行为必须保持不变

按工作模式调整深度：

| 模式 | 摸底深度 |
| --- | --- |
| `patch-lite` | 可压缩，但至少覆盖目标组件的当前行为和直接依赖 |
| `feature-slice` | 完整覆盖目标模块的用户路径、状态源和共享依赖 |
| `migration-strict` | 额外覆盖共享依赖边界、兼容承诺、staged rollout 约束、已有验证缺口 |

退出门禁：

- 已确认"现在是怎样"的证据
- 能说明哪些行为不能被意外改变
- 能说明这次改动最可能伤到哪里

---

### Phase B — 自学习

回答"我已经知道什么"。

读取三层输入：

1. **流程层**：过往 reflect、SDD 沉淀、immune 资产
   - **正式 immune**（`immune-registry.yaml`）：已验证的长期防护（如回归测试、规则、文档化的已知陷阱）→ 硬约束，必须尊重
   - **候选 immune**（`immune-candidates.yaml`）：范围或证据尚不充分的观察 → 弱信号，只提醒不拦截
2. **项目层**：`.project-context`、项目规则、共享模式
3. **模块层**：目标模块架构、入口点、已知风险

对每个关键问题标注三色置信度：

| 状态 | 含义 |
| --- | --- |
| 🔴 | 缺失关键信息，自学习无法回答，必须向用户澄清 |
| 🟡 | AI 基于证据做了推测，展示推理过程，由用户决定是否需要澄清 |
| 🟢 | 通过代码、文档、测试或用户历史回答已确认 |

产出格式：

```markdown
## 自学习状态表

| # | 问题 | 状态 | 依据 / 推理 |
| --- | --- | --- | --- |
| Q-1 | ... | 🔴 / 🟡 / 🟢 | ... |
```

退出门禁：

- 每个关键问题都有状态和依据
- 🟡 项的推理过程已写出
- 🔴 项已标明为什么自学习无法回答

---

### Phase C — 澄清

基于 Phase B 的状态表，与用户进行多轮澄清，直到高优先级问题全部变绿。

**第一步：展示状态表并提示用户**

向用户展示完整状态表，并明确提示：

> 以下 🔴 项必须回答才能继续。🟡 项是我基于现有证据的推测（推理过程已列出），你可以：确认推测、要求澄清、或标记为可接受的工作假设。

**第二步：确定澄清粒度**

澄清粒度控制的是你愿意在分析阶段花多少时间做风险收敛。项目越严肃，应该花越多时间。

| 粒度 | 行为 | 适用场景 |
| --- | --- | --- |
| **全量澄清** | 🔴 + 所有 🟡 逐条澄清 | 线上功能、迁移、高风险改造 |
| **按需澄清** | 🔴 必须澄清，🟡 由用户挑选 | 日常功能开发 |
| **最小澄清** | 只澄清 🔴，🟡 接受为工作假设 | 探索、POC、内部工具 |

如果用户不显式选择，按工作模式默认：

- `migration-strict` → 全量澄清
- `feature-slice` → 按需澄清
- `patch-lite` → 最小澄清

**第三步：多轮收敛**

每轮：

1. 用户回答当前批次问题
2. AI 更新状态表（红→绿、黄→绿，或新信息触发新的红/黄）
3. 检查准出条件

准出条件：

- 所有 🔴 变成 🟢
- 所有 🟡 要么变成 🟢，要么被用户显式接受为工作假设（标记为 `🟡 已接受`，进入规格阶段时计入假设项）
- 如果某轮澄清后没有新增 🔴 且用户不再挑选 🟡，退出

---

### Phase D — 规格

综合 Phase A-C 产出分析包。此时按需读取已激活的领域检查清单（见 `references/` 下的对应文件）。

产出 `analysis-spec.md`，最小输出结构：

```markdown
# Analysis Spec: <topic>

## 现状摸底
<!-- Phase A: 当前行为、入口点、共享依赖、风险点、验证缺口 -->

## 自学习状态表

| # | 问题 | 状态 | 依据 / 推理 |
| --- | --- | --- | --- |
| Q-1 | | 🔴 / 🟡 / 🟢 | |

## 澄清结论
<!-- 记录每轮澄清结果，含用户确认和 🟡 已接受项 -->

## 激活的领域检查清单

- [ ] 前端
- [ ] 后端
- [ ] 集成
- [ ] 风险记忆

## 目标行为

| # | 行为 | 验收标准 |
| --- | --- | --- |
| TB-1 | | |

## 不做的事

## 风险清单

| 风险 | 严重度 | 缓解方式 |
| --- | --- | --- |

## 证据计划

| TB # | 验证方式 |
| --- | --- |
```

退出门禁：

- 每条目标行为可验证
- 无"实现时再看"的模糊表述
- 延后项已显式标记
- 证据计划与目标行为一一对应

## 领域检查清单

Phase D 执行时，只读取已激活的检查清单文件，不需要全部加载。

### 内置默认

| 检查清单 | 文件 |
| --- | --- |
| 前端 | [references/frontend-lens.md](references/frontend-lens.md) |
| 后端 | [references/backend-lens.md](references/backend-lens.md) |
| 集成 | [references/integration-lens.md](references/integration-lens.md) |
| 风险记忆 | [references/risk-memory-lens.md](references/risk-memory-lens.md) |

### 自定义与扩展

检查清单的自定义和扩展都在仓库的 `.project-context/lenses/` 目录下完成，不需要修改 skill 本身。

**覆盖默认检查清单**：复制 `references/` 下的同名文件到 `.project-context/lenses/`，修改内容。Phase D 执行时，同名文件优先使用仓库版本。

**新增领域检查清单**：在 `.project-context/lenses/` 下新建 `.md` 文件。Phase D 执行时自动扫描该目录下所有 `.md` 文件，识别为可选检查清单。

格式模板见 [references/lens-template.md](references/lens-template.md)。

**优先级**：`.project-context/lenses/` > `references/`（同名覆盖）。仓库版本永远优先于 skill 内置默认。

### 设计边界

自定义 lens 的目标，是补充默认检查清单覆盖不够、但会在这个产品多个特性中反复出现的"特性级校验维度"。它不是某个单一需求的验收清单，也不是泛泛的质量口号。

建议约束：

- 一个项目默认新增 `0-3` 个自定义 lens。没有明显项目差异时，直接沿用内置默认。
- 每个 lens 保持 `4-8` 条检查项。超过 `8` 条通常说明主题混杂，应该拆分或回退到默认 lens。
- 每条检查项应同时满足：跨多个特性反复出现、漏掉会造成验收缺口或产品风险、能在 `analysis-spec` 中被具体回答、默认 lens 覆盖不够。
- 优先写工作流、可见性 / 权限、通知 / 审计、副作用边界、租户边界这类产品校验维度。
- 不要写页面细节、一次性需求规则、实现方案，或"体验是否良好""注意性能"这类空泛表述。
- 如果一条检查项已经被 `frontend` / `backend` / `integration` / `risk-memory` 稳定覆盖，不要在项目 lens 中重复。
- lens 本体只保留已确认、可长期复用的检查项；不要混入 `draft` 时间、`待确认`、`assumption` 等过程信息。
- 初始化阶段的项目 lens 应通过引导式提问从用户处提炼，而非 AI 单方面推断后让用户确认。详见 `initialize` Phase 3.5。

### Good Case / Bad Case

Good case：

- `workflow.md`：适用于审批、状态机、任务流类产品，检查项围绕状态流转闭环、角色动作边界、撤回 / 超时、通知副作用、操作审计。
- `tenant-visibility.md`：适用于多租户或多角色产品，检查项围绕数据可见范围、跨角色一致性、导出 / 分享边界、越权风险。
- `audit-notify.md`：适用于关键操作需要留痕和消息触达的产品，检查项围绕谁触发、何时记录、谁接收通知、失败如何补偿。

Bad case：

- `order-detail-page.md`：只描述某个页面的按钮状态、弹窗交互、列表样式。这是单次需求的目标行为，不该上升为项目 lens。
- `quality.md`：只有"用户体验是否良好""代码是否可维护""性能是否要注意"之类表述，太空泛，无法在分析阶段给出稳定答案。
- `misc.md`：把工作流、移动端、埋点、缓存、文案一次塞进同一个 lens，主题过散，后续只会变成走过场。
- `content-integrity.md` 这类文件如果混入 `draft: ...`、`待确认` 小节、或一次初始化才需要看的临时疑问，也说明过程信息泄漏进了 lens 本体，应该在落盘前先收敛。

## 工件管理

| 工件 | 路径约定 |
| --- | --- |
| `analysis-spec.md` | `docs/specs/<topic>/analysis-spec.md` |

`<topic>` 使用需求的短标识，kebab-case，如 `order-list-pagination`。

后续工件（`slice-plan.md`、`verify.md`、`spec-check.md`、`reflect.md`）都放在同一个 `docs/specs/<topic>/` 下。多 slice 需求的 per-slice 工件放在 `docs/specs/<topic>/slices/<slice-id>/`。

`patch-lite` 可以把 analysis-spec、slice-plan、verify 合并成一份精简记录，但 spec-check / reflect 语义不可跳过。

---

## 降级模式

如果 `.project-context` 缺失或过期：

- Phase A 以代码和直接证据为准
- Phase B 跳过流程层和项目层输入，只做模块层
- 在最终输出中标注上下文缺口，建议后续补 `initialize` 或 `sync-context`

如果用户拒绝进入澄清：

- 所有 🔴 标记为 `unresolved`
- 所有 🟡 标记为 `assumed`
- 在 analysis-spec.md 中显式声明这些风险

## 最终输出

结束时必须包含：

### 分析摘要

- 工作模式
- 激活的领域检查清单
- 澄清粒度和轮次
- 关键风险

### 产出文件

- `analysis-spec.md` 路径

### 下一步

- 建议进入 `slice-plan`
- 如果发现需要升级工作模式，建议回到 Phase 0 重新分级
