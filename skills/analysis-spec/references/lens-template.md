## 设计边界

- 只保留已确认、可长期复用的检查项。
- 只记录跨多个产品特性反复出现的校验维度。
- 默认保持 `4-8` 条检查项；超过 `8` 条时，优先考虑拆分。
- 优先补充默认 `frontend` / `backend` / `integration` / `risk-memory` 覆盖不够的部分。
- 不要写单页面细节、一次性需求、实现方案，或空泛口号。
- 不要在 lens 文件中写 `draft` 时间、`待确认`、`assumption` 等过程性信息。

## Good Case

```markdown
# workflow

适用信号：需求涉及状态流转、角色审批、撤回 / 驳回、通知副作用。

## 检查项

- [ ] 状态流转是否闭环，是否存在不可恢复的中间态
- [ ] 不同角色各自能执行哪些动作，是否存在越权或漏权
- [ ] 撤回、驳回、超时、重试等逆向路径是否明确
- [ ] 每个关键动作是否伴随正确的通知、副作用或后续任务
- [ ] 是否需要可追溯的操作审计
```

## Bad Case

- `order-detail-page`：只描述订单详情页按钮、弹窗、文案和排序规则，过于贴近单一特性，应该写进当次 `analysis-spec` 的目标行为。
- `quality`：只有"体验是否良好""代码是否可维护""注意性能"之类表述，过于空泛，难以回答和验证。
- `misc`：把审批、移动端、埋点、缓存和导出限制全塞进一个 lens，主题过散，应该拆分或回退到默认 lens。
- `content-integrity`：文件里混入 `draft: 2026-03-27`、`待确认` 小节或一次性初始化疑问，说明还没收敛完成，不应该直接作为正式 lens 落盘。

## 使用方式

1. 复制本文件到 `.project-context/lenses/<你的检查清单名>.md`
2. 替换上面的占位内容
3. 下次执行 `analysis-spec` Phase D 时会自动识别并提供激活选项
