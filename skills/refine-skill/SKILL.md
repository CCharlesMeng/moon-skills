# Refine Skill

**显式调用**:`/refine-skill <skill-name>`,目标就是本次会话里被执行/调试的那个 skill。

一个 skill 是关于"怎么做这类任务"的可证伪理论,不是操作清单。本次会话里每次纠正和返工,都是对这个理论的反驳;复盘是修理论,不是给每个错糊补丁。整个过程五步,依次走完,不跳步。

## 第一步:收集证据

两路证据都自己拿——委派的判据不是活重不重,而是**摘要是否无损**,而这两样恰恰经不起摘要:

- **会话证据只有你有。** 本次会话的每次纠正与返工都在你的上下文里,自己盘点,逐条记成 `触发情境 → 错误行为 → 用户想要的正确行为`。
- **目标 skill 全目录自己通读**(`SKILL.md`、`references/`、`scripts/`、`evals/`)。原文就是证据:找被反驳的假设、判紧松、守住改写的连贯性,都依赖逐字措辞,子代理的摘要会把这个信号毁掉;何况一个 skill 通常就几 K token,读它的成本可以忽略。真正该委派的是后面的重放与打分——机械、量大、摘要无损。

## 第二步:判断

五条依据,每条是一个先于"怎么改"的问题。

**一、找那条被反驳的假设。** 先别问"该加哪句话",先问:skill 当时让 agent 相信了什么,它才这么做?散落的错往往是同一个错误假设的不同表现。找那条改掉就能让一整类错消失的根本误解——改一处,胜过补五处。

**二、分流:系统性,还是一次性偶然。** 一次会话是 n=1 的证据,据此改一条承重规则就是过拟合。偶然(模型抽风、罕见输入、外部故障)不动规则,只把情境追加进 `evals/` 观察;复发了,它自己会升级成系统性。只有系统性失败才值得改规则。

**三、太紧还是太松。** 系统性错误只来自两个方向:skill 管太死把 agent 逼上错路,或该替它定的没定、放任它瞎猜。先答这个,再决定收紧、放宽还是删除。默认往上加步骤,通常是错的方向。

**四、执行经济性。** 主 agent 的上下文是整个系统里最贵的资源。检查目标 skill:有没有让主 agent 亲自做**摘要无损**的重活(批量重试、机械比对、按固定判据打分)?这些都是委派候选。反之,产出依赖逐字原文或全局判断的工作,委派只会毁掉信号。凡改成委派,提示词必须同时升级成契约(见第四步),否则只是把错误换了个便宜的地方犯。

**五、用更短换更对。** 想加步骤时停一下:有没有一条原则能让 agent 自己推出它?有就加原则。好的改写通常让 skill 更短或持平——删掉 agent 本就会的指令,和新增一样是改进。多次复盘后,守住单一连贯的设计意图,别让它变成补丁的拼盘。

## 第三步:与用户确认

给一份判断,不是一份问卷:根本误解是什么、系统性还是偶然、太紧还是太松、有无经济性问题、打算怎么改(改前/改后对照)。**确认后再动文件。**

## 第四步:改写

改写时两把尺子,对应指令的两种身份:

- **判断写成原则,执行写成契约。** 需要权衡的内容(改不改、何时改)用原则表达,让强模型能外推到没写到的情境;交给子代理执行的内容写成自包含契约——**输入材料、输出格式、完成判据**三要素齐备,不依赖会话记忆,不留"自行判断"的空隙。skill 本身可能整个跑在能力一般的模型上,所以流程骨架也按契约标准写:显式步骤,无歧义。
- **引用必须承重。** 理论引用只保留改变 agent 行为的那种:删掉这句话行为会变差,才留;不会变差的是装饰,删。行文以人一遍读懂为准,不炫技。

## 第五步:回归验证

prose 没法像代码那样断言对错,但能做概率性回归测试。**不要在写出改法的这个会话里验证——agent 已经知道答案,等于拿训练集当测试集。**

1. 把本次每条系统性错误写成新用例,追加进目标 skill 的 `evals/evals.json`(没有就创建);再补 1-2 个本次**做对了**的行为。格式与维护规则见 `references/evaluation.md`。
2. 委派干净子代理(不带本对话记忆)重放**全套**用例,不只是本次新增的——上上次修好的错,只有全量重放才保证不被这次改动翻出来。用实际使用档位的模型跑,每用例约 3 次取多数。
3. 通过线两条:新用例不再复发,旧用例仍然全过。旧用例退化才是"质量下降"的真正来源,通常意味着收太紧,往回退再验。
4. 让 grader 顺带批判断言本身:永远会过的断言、没被任何断言覆盖的重要行为,都要报上来。评测标准和 skill 一起进化。

## 收口

不写 CHANGELOG——evals 是测试集,git 历史是变更记录,各归其位。最后给用户一句明确的推荐下一步,从这些里选最合适的:

- 验证通过 → 建议提交(给出简短 commit message),并在接下来几次真实使用中留意是否复发。
- 验证不充分(用例太少 / 非确定性大)→ 建议再积累几次会话证据,暂不改承重规则。
- 错误判定为偶然 → 不改 skill,用例已入 `evals/` 观察。
- 改写触及自由度刻度 → 提醒用户这类改动影响面大,建议小步改、多观察。
