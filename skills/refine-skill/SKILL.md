# Refine Skill

**显式调用**:`/refine-skill <skill-name>`,目标就是本次会话里被执行/调试的那个 skill。

一个 skill 是关于"怎么做这类任务"的一个可证伪的理论,不是操作清单。本次会话里每次纠正和返工,都是对这个理论的反驳。复盘是修理论,不是给每个错糊补丁。四条判断依据如下,具体怎么改从中自己推。

## 一、找那条被反驳的假设

每次纠正或返工,先别问"该加哪句话",先问:**skill 当时让 agent 相信了什么,它才这么做?** 散落的错往往是同一个错误假设的不同表现。找那条改掉就能让一整类错消失的根本误解——改一处,胜过补五处。

## 二、先分流:系统性,还是一次性偶然

不是每个错都该写进 skill。一次会话是 n=1 的证据,据此改一条承重规则就是过拟合。先判断:这是任务里**反复会犯**的系统性失败,还是这一次的偶然(模型抽风、罕见输入、外部故障)?偶然只记下,不动 skill。只有系统性失败才值得改。

## 三、判断太紧还是太松

系统性错误只来自两个方向:skill 管太死把 agent 逼上错路,或该替它定的没定、放任它瞎猜。每条先回答这一个问题,再决定收紧、放宽,还是删除。默认往上加步骤,通常是错的方向。

## 四、用更短换更对

想加步骤时停一下:有没有一条原则能让 agent 自己推出它?有就加原则。好的改写通常让 skill 更短或持平,极少更长——删掉 agent 本就会的指令和新增一样是改进。多次复盘后,守住单一连贯的设计意图,别让它变成一堆补丁的拼盘。

## 验证:别让改完反而更差

prose 没法像代码那样断言对错,但能做概率性回归测试。**关键:不要在写出改法的这个会话里验证——agent 已经知道答案了,等于拿训练集当测试集。必须在干净上下文里重放。**

1. 把每条系统性错误变成一个用例:`触发情境 → 用户当时想要的正确行为`。再挑 1-2 个本次**做对了**的行为。
2. 开一个全新子代理(`Task` 工具 / Cursor SDK,不带本对话记忆),只给它改后的 skill + 触发情境,看它怎么做。每个用例跑约 3 次取多数。
3. 两条通过线:错误用例**不再复发**;做对的用例**仍然做对**。后者退化才是"质量下降"的真正来源。
4. 若某个原本对的用例退化了 → 改写过头(通常收太紧),往回退再验。

## 落地与收口

读全目标 skill 目录(`SKILL.md` 及 reference/scripts);先给用户一份判断——根本误解是什么、属系统性还是偶然、太紧还是太松、打算怎么改(改前/改后),**确认后再动文件**;改后按上面验证;不写 CHANGELOG,改动只留在指令和 git 历史里。

最后给用户一句明确的推荐下一步,从这些里选最合适的:
- 验证通过 → 建议提交(给出简短 commit message),并在接下来几次真实使用中留意是否复发。
- 验证不充分(用例太少 / 非确定性大)→ 建议再积累几次会话证据,暂不改承重规则。
- 错误判定为偶然 → 建议不改 skill,仅记录现象。
- 改写触及自由度刻度 → 提醒用户这类改动影响面大,建议小步改、多观察。
