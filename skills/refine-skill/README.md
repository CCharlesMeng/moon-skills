# refine-skill

在**同一会话末尾**复盘某个 skill 本次执行出的错,改写它,让它下次不再重犯或变得更好用。

核心理念:一个 skill 不是操作清单,而是一组关于"这类任务该怎么做"的假设。会话里每次被你纠正、返工,都是现实对某条假设的一次反驳。refine-skill 做的是**修假设**,而不是给每个错糊一个补丁。

## 适用场景

- 刚用某个 skill 跑完一个任务,过程中纠正了它好几次,想让这些教训沉淀下来;
- 某个 skill 反复在同一类事情上出错,想找到根本原因;
- 想给 skill 补回归测试(evals),防止改好的东西以后又坏回去。

**不适用**:从零创建新 skill(用 `write-a-skill` / `create-skill`);复盘的错误发生在别的会话里(证据只在当前会话的上下文中,跨会话拿不到)。

## 如何调用

前提:在**执行/调试过目标 skill 的那个会话里**调用,不要新开会话。

### 方式一:显式命令(推荐)

```
/refine-skill <skill-name>
```

### 方式二:自然语言

以下说法都会触发:

- "复盘一下 xxx 这个 skill"
- "改进/优化 xxx skill,避免重犯刚才的错"
- "总结一下 xxx 这次出的错,把它改好"

## Prompt 示例

**基础用法** —— 会话末尾直接调用:

```
/refine-skill hocb-monitor
```

**带补充观察** —— 把你注意到的问题一并交代,归因会更准:

```
/refine-skill research

补充两点:一是它把报告写成了英文,我要的是中文;二是引用的两个来源
其实是二手转述,不是我要求的一手来源。第一点可能是我没说清楚,
第二点我觉得是它自己判断有问题,你分析看看。
```

**只做归因分析,先不动文件**:

```
复盘一下本次 tdd skill 的执行:把所有被我纠正的内容逐条列出来,
判断每条是信息不足还是判断错误,用表格给出修改内容、错误归因、
处置建议。先不要改任何文件,等我确认。
```

**怀疑是偶发,想观察而不是改规则**:

```
/refine-skill diagnose

刚才那个 JSON 写成 YAML 的错,我怀疑是模型抽风不是 skill 的问题,
你判断一下,如果是偶然就只记入 evals 观察,别加规则。
```

## 它会做什么(五步流程)

1. **收集证据**:盘点本次会话里所有纠正与返工(`触发情境 → 错误行为 → 正确行为`),并通读目标 skill 全目录原文。
2. **判断**:对每条纠正依次过六个问题——
   - **先归因**:信息不足还是判断错误?任务特有的信息缺口不改 skill,转为"下次指令建议";
   - **找被反驳的假设**:skill 让 agent 相信了什么才犯的错,改一处胜过补五处;
   - **分流**:系统性错误才改规则,一次性偶然只记入 evals 观察;
   - **太紧还是太松**:是管得太死逼出的错,还是该定的没定;
   - **执行经济性**:有没有该委派给子代理的重活占用了主 agent 上下文;
   - **用更短换更对**:能用一条原则推出的就不加步骤。
3. **与用户确认**:给出归因表格(`纠正内容 | 归因 | 处置`)和整体判断、改前/改后对照,**你确认后才动文件**。
4. **改写**:判断写成原则,执行写成契约;删掉不承重的装饰性内容。
5. **回归验证**:把本次错误写成新用例追加进目标 skill 的 `evals/evals.json`,委派干净子代理全量重放验证——新错不复发、旧用例不退化。注意:重放验证不在改写的同一会话做(agent 已知答案,等于拿训练集当测试集)。

## 输出是什么

- 一份归因表格 + 判断结论(第三步,供你确认);
- 改写后的目标 skill 文件(经你确认后);
- 追加了新用例的 `evals/evals.json`(在目标 skill 目录内);
- 收口时的下一步建议:建议提交 / 再积累证据 / 仅观察不改 / 下次指令的补充信息。

不写 CHANGELOG——evals 是测试集,git 历史是变更记录,各归其位。
