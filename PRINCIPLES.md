# Skill-User 交互黄金原则

面向本仓库所有交互式 skill 的输出规范。每次编写或优化 skill 时，必须对照这 7 条原则检查交互行为。

---

## P1 — 单轮单决策

每一轮 AI 输出只包含 **一个决策点**：

- 要么请求确认
- 要么提问（批量，见 P7）
- 要么宣告完成并给出下一步索引（见 P5）

同一轮不得同时有确认门 + 问题 + 路由三件事。

---

## P2 — 结构优于散文

所有多项内容必须用表格或列表，不写段落。

- 纯散文只出现在单行摘要，不超过 2 句
- 模糊邀请语（"如果有问题现在提"）必须替换为明确行动指令

**不可以：**
```
以上是你最终会得到的东西。如果和你的预期不一致，现在是最好的纠正时机。
```

**应该：**
```
→ 请确认继续 / 或指出需要修改的地方。
```

---

## P3 — 不解释过程，直接呈现结果

禁止以下开场：
- "现在我将开始…"
- "按照 skill 的要求，我会…"
- "接下来进入 Phase X…"

每轮直接输出结果，不解释即将要做的事。

---

## P4 — 确认门格式标准化

所有有确认门的 Phase，统一用以下格式结束：

```
---
**[Phase N 确认门]** <本阶段产出的一句话描述>
→ 请确认继续 / 或指出需要修改的地方。
---
```

不允许用模糊邀请语替代。不允许在确认门之后追加"如果…可以…"的补充说明。

---

## P5 — 最终输出是三行索引，不是报告

skill 执行完成时，只输出：

```
✓ <skill 名> 完成
产出：<文件路径>
下一步：<一条具体指令>（附触发条件，如适用）
```

不得重新摘要刚才交互过的内容。不得写多个子章节（"设计摘要"、"产出文件"、"下一步"）分开描述。

---

## P6 — 零决策阶段自动内联

如果某个 Phase 的结果用户不需要做任何决策（典型：工作模式判断、上下文预热），将其内联进下一个有决策的 Phase，不单独产出一轮输出。

**例外**：当结论出乎意料（如工作模式升级为 `migration-strict`），才单独一行告知。

---

## P7 — 批量提问，不逐条打散

澄清或提问时，同一优先级的问题必须合并在一轮，不拆成多轮一个个问。

格式：

```
以下 N 个问题需要你回答：

| # | 问题 | 我的推测（如有） |
|---|------|-----------------|
| Q-1 | ... | — |
| Q-2 | ... | 推测为 X，理由：... |

🔴 标记的问题必须回答才能继续；🟡 标记的是推测，可确认、纠正或跳过（标为工作假设）。
```

不使用段落文字引导用户回答，不在每个问题后写单独解释。

---

## 应用方式

在 skill 的 SKILL.md 中，用 `## 交互规范` 章节声明本 skill 遵守的原则，并补充 skill 专属的每 Phase 输出格式表：

```markdown
## 交互规范

遵守 [Skill-User 交互黄金原则](../../PRINCIPLES.md)。

| Phase | 用户可见内容 | 结束方式 |
| --- | --- | --- |
| 0 | 内联进 Phase A（P6） | — |
| A | ... | 确认门（P4） |
| B | ... | — / 确认门（P4） |
| 最终 | 三行索引（P5） | — |
```
